## 一、功能概述：为什么需要 buildProgress？

### 痛点

在大型 Vite 项目中，运行 `vite build` 后终端常常长时间没有任何输出。你无法判断：

- 构建是正在进行还是已经卡死？
- 当前处于构建的哪个阶段？
- 还需要等多久才能完成？

这种"黑盒"体验不仅影响开发效率，在 CI/CD 流水线中更难以追踪构建状态。

### 解决方案

`buildProgress` 插件通过监听 Vite 构建生命周期的 9 个钩子，在终端实时显示：

- **进度百分比** — 基于构建阶段和模块转换比例精确计算
- **阶段标签** — "读取配置"、"解析模块"、"转换模块"、"打包中"、"写入文件"
- **可视化进度条** — 用填充字符直观展示完成比例
- **模块名称** — 显示当前正在处理的文件，快速定位瓶颈

```
⠋ 转换模块 ██████████████████████░░░░░░░░ 67% src/components/App.vue
```

### ProgressTheme 接口

```typescript
interface ProgressTheme {
	completeColor: (text: string) => string // 已完成部分颜色
	incompleteColor: (text: string) => string // 未完成部分颜色
	percentageColor: (text: string) => string // 百分比数字颜色
```

## 七、技术实现深度解析

### 7.1 进度计算模型

插件的进度计算基于 Vite 构建生命周期的阶段划分：

```
┌──────────┬──────────────┬──────────────────────────────────────┐
│   阶段    │   进度范围    │   说明                                │
├──────────┼──────────────┼──────────────────────────────────────┤
│ config   │ 5%           │ 读取配置阶段                          │
│ resolve  │ 10%          │ 解析模块依赖阶段                      │
│ transform│ 15% - 85%    │ 转换模块阶段（按模块比例线性计算）      │
│ bundle   │ +10%         │ 打包阶段（仅生产构建）                │
│ write    │ +5%          │ 写入文件阶段                          │
│ done     │ 100%         │ 构建完成                              │
└──────────┴──────────────┴──────────────────────────────────────┘
```

`transform` 阶段占据最大区间（70%），因为它是构建中最耗时的环节。进度按已转换模块数与总模块数的比例线性计算：

```typescript
private getPercentage(): number {
  if (this.phase === 'done') return 100
  if (this.phase === 'config') return 5
  if (this.phase === 'resolve') return 10
  if (this.totalModules === 0) return 15

  const transformProgress = Math.min(
    (this.transformedModules / this.totalModules) * 70, 70
  )
  const bundleProgress = this.phase === 'bundle' ? 10 : 0
  const writeProgress = this.phase === 'write' ? 5 : 0

  return Math.min(Math.floor(15 + transformProgress + bundleProgress + writeProgress), 99)
}
```

**关键设计：进度只进不退**。通过 `lastPercentage` 缓存上一次计算的值，确保进度数值单调递增：

```typescript
this.lastPercentage = Math.max(calculated, this.lastPercentage)
```

这避免了因模块计数波动导致进度条"倒退"的视觉闪烁。

### 7.2 Vite 钩子映射

插件监听了 9 个 Vite 钩子，覆盖构建全流程：

```
┌─────────────────────┬──────────────────────────────────────┐
│       Vite 钩子      │              插件行为                 │
├─────────────────────┼──────────────────────────────────────┤
│ config              │ 识别开发/生产环境                      │
│ configResolved      │ 初始化进度显示，隐藏终端光标            │
│ buildStart          │ 重置模块计数器                         │
│ resolveId           │ 统计模块总数（排除 node_modules）       │
│ transform           │ 追踪转换进度，记录当前模块名            │
│ buildEnd            │ 标记打包阶段（仅生产构建）              │
│ writeBundle         │ 标记写入阶段                           │
│ closeBundle         │ 完成进度显示                           │
│ configureServer     │ 开发服务器就绪时完成进度（仅开发模式）   │
└─────────────────────┴──────────────────────────────────────┘
```

**模块过滤策略**：`resolveId` 和 `transform` 钩子中排除了 `node_modules` 和 `.virtual` 模块，只统计项目源码模块，确保进度百分比反映实际的业务代码转换进度。

### 7.3 Spinner 动画机制

动画通过 `setInterval` 以 80ms 间隔定时刷新终端输出：

```typescript
private startSpinner(): void {
  if (this.spinnerTimer) return
  if (!process.stdout.isTTY) return
  this.spinnerTimer = setInterval(() => this.update(), 80)
}
```

**跨平台字符选择**：

```typescript
// Windows 使用 ASCII 字符
export const SPINNER_FRAMES = process.platform === 'win32' ? ['|', '/', '-', '\\'] : ['⠋', '⠙', '⠹', '⠸', '⠼', '⠴', '⠦', '⠧', '⠇', '⠏']
```

### 7.4 终端状态管理

插件在构建过程中会修改终端状态，必须确保正确恢复：

| 时机             | 操作         | ANSI 序列   |
| ---------------- | ------------ | ----------- |
| `configResolved` | 隐藏光标     | `\x1b[?25l` |
| `complete()`     | 恢复光标     | `\x1b[?25h` |
| `update()`       | 清除当前行   | `\x1b[2K`   |
| `update()`       | 光标移至行首 | `\x1b[0G`   |

`destroy()` 方法作为安全网，确保即使构建异常中断也能恢复终端状态：

```typescript
protected destroy(): void {
  super.destroy()        // 注销日志配置
  this.stopSpinner()     // 停止动画定时器
  if (process.stdout.isTTY) {
    process.stdout.write(ANSI.showCursor)  // 恢复光标
  }
}
```

### 7.5 非 TTY 降级策略

在 CI/CD 等 non-TTY 环境中，`process.stdout.isTTY` 为 `undefined`：

```
┌──────────────────┬────────────────────────────────────┐
│     TTY 环境      │           非 TTY 环境               │
├──────────────────┼────────────────────────────────────┤
│ 实时进度条渲染    │ 跳过所有终端写入操作                │
│ Spinner 动画     │ 不启动定时器                        │
│ 光标隐藏/恢复    │ 不修改终端状态                      │
│ 构建完成时       │ 通过 Logger 输出 "构建完成" 日志     │
└──────────────────┴────────────────────────────────────┘
```

无需任何额外配置，插件自动适配运行环境。

## 八、进阶用法：自定义外观与主题

### 自定义进度条字符

```typescript
buildProgress({
	width: 40,
	completeChar: '■',
	incompleteChar: '□',
	clearOnComplete: false
})
```


`clearOnComplete: false` 让构建完成后保留 100% 进度条在终端中，方便确认构建成功。

### 自定义颜色主题

使用 ANSI 颜色码为进度条各部分着色：

```typescript
buildProgress({
	theme: {
		completeColor: t => `\x1b[32m${t}\x1b[39m`, // 绿色 — 已完成部分
		incompleteColor: t => `\x1b[90m${t}\x1b[39m`, // 灰色 — 未完成部分
		percentageColor: t => `\x1b[1m${t}\x1b[22m`, // 粗体 — 百分比数字
		phaseColor: t => `\x1b[36m${t}\x1b[39m`, // 青色 — 阶段标签
		moduleColor: t => `\x1b[33m${t}\x1b[39m` // 黄色 — 模块名称
	}
})
```

也可以使用 `chalk`、`picocolors` 等库简化 ANSI 码编写：

```typescript
import pc from 'picocolors'

buildProgress({
	theme: {
		completeColor: pc.green,
		incompleteColor: pc.gray,
		percentageColor: pc.bold,
		phaseColor: pc.cyan,
		moduleColor: pc.yellow
	}
})
```

### 条件启用

根据环境变量控制插件的启用：

```typescript
buildProgress({
	enabled: process.env.CI !== 'true' // CI 环境禁用进度条
```

## 十、注意事项

### 1. 终端兼容性

- 进度条依赖 ANSI 转义码，需要终端支持。大多数现代终端（iTerm2、Windows Terminal、VS Code 内置终端）均支持
- 部分 IDE 的旧版内置终端可能不支持光标隐藏，导致 `^[[?25l` 等转义序列可见
- 非 TTY 环境（CI/CD、重定向输出）自动降级，无需担心

### 2. 构建中断后的终端恢复

如果构建被 `Ctrl+C` 强制中断，`destroy()` 可能未被调用，导致终端光标消失。恢复方法：

```bash
# 方法 1：重置终端
reset

# 方法 2：手动发送恢复序列
echo -e "\x1b[?25h"
```

### 3. 进度百分比的特点

- 进度**只进不退**，这是有意为之的设计，避免视觉闪烁
- `transform` → `bundle` 阶段之间，进度可能在 85%-90% 区间停留较长时间，这是正常的——Rollup 打包的耗时与模块数量无线性关系
- 进度最高显示 99%，只有构建真正完成时才显示 100%

### 4. 与其他插件的执行顺序

`buildProgress` 建议放在插件列表的前部，确保其 `configResolved` 和 `closeBundle` 钩子能覆盖完整的构建过程：

```typescript
export default defineConfig({
  plugins: [
    buildProgress(),    // ✅ 建议靠前
    copyFile(...),
    generateVersion(...),
    injectIco(...)
  ]
})
```

### 5. 开发模式下的行为

开发模式（`vite dev`）下，插件在 `configureServer` 钩子触发时完成进度显示。由于开发服务器启动较快，进度条可能一闪而过，这是正常现象。

## 十一、常见问题解答

### Q1：构建时没有看到进度条？

**排查步骤**：

1. 确认 `enabled` 未设为 `false`
2. 确认终端支持 TTY — 在终端中运行 `node -e "console.log(process.stdout.isTTY)"`，应输出 `true`
3. 确认 `vite.config.ts` 中已正确引入插件
4. 如果使用 `format: 'bar'`，确认终端宽度足够（建议 ≥ 60 列）

### Q2：进度条显示乱码？

可能是终端不支持 Unicode Braille 字符。解决方案：

```typescript
buildProgress({
	completeChar: '=',
	incompleteChar: '-'
})
```

Windows 用户如果看到乱码，可以尝试使用 Windows Terminal 替代 CMD。

### Q3：CI/CD 中如何使用？

无需特殊配置。插件自动检测 non-TTY 环境并降级为日志输出：

```
[@meng-xi/vite-plugin:build-progress] ✔ 构建完成
```

如果希望完全禁用（避免任何输出），设置 `enabled: false` 或通过环境变量控制：

```typescript
buildProgress({
	enabled: process.env.CI !== 'true'
})
```

### Q4：如何为不同环境使用不同的格式？

```typescript
buildProgress({
	format: process.env.CI ? 'minimal' : 'bar'
})
```

### Q5：进度条和 Vite 自带的构建输出冲突吗？

不冲突。`buildProgress` 使用 `process.stdout.write` 直接写入终端，而 Vite 的构建输出使用 Logger。两者独立工作。如果希望更清爽的输出，可以设置 `clearOnComplete: true`（默认值），构建完成后进度条会被清除。

### Q6：可以在多个项目中复用自定义主题吗？

可以。将主题配置提取为常量：

```typescript
// themes.ts
export const myTheme = {
	completeColor: (t: string) => `\x1b[32m${t}\x1b[39m`,
	incompleteColor: (t: string) => `\x1b[90m${t}\x1b[39m`,
	percentageColor: (t: string) => `\x1b[1m${t}\x1b[22m`,
	phaseColor: (t: string) => `\x1b[36m${t}\x1b[39m`,
	moduleColor: (t: string) => `\x1b[33m${t}\x1b[39m`
}

// vite.config.ts
import { myTheme } from './themes'
buildProgress({ theme: myTheme })
```

### Q7：buildProgress 支持 dev 模式吗？

支持。开发模式下，插件在 `configureServer` 钩子（即开发服务器就绪后）自动完成进度显示。由于开发服务器启动通常很快，进度条可能一闪而过。

## 十二、总结与展望

### v0.0.7 核心价值

`buildProgress` 插件为 `@meng-xi/vite-plugin` 带来了构建过程的可视化能力，填补了 Vite 生态中构建进度反馈的空白。它的设计遵循以下原则：

- **零配置可用** — `buildProgress()` 一行代码即可启用
- **灵活可定制** — 三种格式、自定义字符、颜色主题，满足不同偏好
- **环境自适应** — TTY/非 TTY 自动切换，无需手动处理
- **框架一致性** — 完全遵循 BasePlugin 规范，与现有插件体验统一
