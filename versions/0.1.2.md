## 写在前面

v0.1.2 是 `@meng-xi/vite-plugin` 的一个重要里程碑版本。这个版本带来了三个核心变化：

1. **第九个插件 `compressAssets`** — 构建产物压缩，支持 gzip / brotli / both 三种算法
2. **零配置体验** — 所有插件参数改为选填，开箱即用
3. **工具层完善** — `@common/object`、`@common/fs`、`@common/format` 全面增强

如果你正在使用 v0.1.1，升级到 v0.1.2 无需任何修改——所有变更向后兼容。如果你是第一次接触这个库，这篇文章将从真实开发场景出发，展示 9 个插件如何精准解决前端工程化痛点。

## 二、从问题出发：compressAssets 解决了什么

### 痛点：构建产物体积大，静态资源传输慢

**场景**：你的 Vite 项目构建完成后，`dist/` 目录下有大量 JS、CSS、HTML、SVG 文件。部署到服务器后，浏览器需要逐个下载这些未压缩的文件，首屏加载时间长，用户体验差。

你可能会在 Nginx 层面开启 gzip，但这有几个问题：

- Nginx 动态压缩消耗服务器 CPU
- CDN 边缘节点可能不支持动态压缩
- 无法同时提供 gzip 和 brotli 两种格式
- 缺乏压缩效果的量化数据

### 解法：compressAssets

```typescript
import { compressAssets } from '@meng-xi/vite-plugin'

export default defineConfig({
	plugins: [
		compressAssets({
			algorithm: 'both',
			threshold: 2048,
			reportOutput: 'compress-report.json'
		})
	]
})
```

**工作原理**：

```
Vite 构建完成 (writeBundle)
    ↓
扫描 dist/ 目录
    ↓
过滤文件（threshold / includeExtensions / excludeExtensions / excludePaths）
    ↓
并发压缩（Worker Pool 模型，parallelLimit 控制并发数）
    ↓
生成 .gz / .br 文件
    ↓
可选：删除原始文件
可选：生成压缩报告
输出压缩率 Top 5 日志
```

### 三种算法选择

| 算法     | 输出后缀      | 适用场景                   | 压缩率 | 压缩速度 |
| -------- | ------------- | -------------------------- | ------ | -------- |
| `gzip`   | `.gz`         | 通用场景，所有浏览器支持   | 中等   | 快       |
| `brotli` | `.br`         | 现代浏览器，追求极致压缩率 | 高     | 慢       |
| `both`   | `.gz` + `.br` | 同时兼容新旧浏览器         | 最高   | 最慢     |

> Nginx 配置 `gzip_static on` 和 `brotli_static on` 后，会自动选择预压缩文件，无需动态压缩。

### 文件过滤优先级

```
1. threshold — 小于阈值的文件跳过
2. excludeExtensions — 排除的扩展名（优先级最高）
3. includeExtensions — 包含的扩展名（列表非空时生效）
4. excludePaths — 排除的路径前缀
5. 已压缩格式 — .gz / .br 自动跳过
```

`reportOutput` 生成的 JSON 报告结构：


## 四、零配置体验：参数可选化详解

v0.1.2 的一个重要设计决策：**所有有默认值的参数均改为选填**。这意味着你可以用最少的配置启动任何插件。

### 4.1 设计原则

```
必填参数 → 用户必须提供，无合理默认值（如 sourceDir、targetDir）
选填参数 → 插件提供合理默认值，用户按需覆盖
```

### 4.2 各插件零配置能力

| 插件                                 | 零配置调用 | 默认行为                                       |
| ------------------------------------ | ---------- | ---------------------------------------------- |
| `buildProgress()`                    | ✅         | 30 字符进度条，bar 格式，完成后清除            |
| `compressAssets()`                   | ✅         | gzip 压缩，1KB 阈值，10 并发，生成报告         |
| `copyFile({ sourceDir, targetDir })` | ⚠️         | 需提供源目录和目标目录                         |
| `faviconManager()`                   | ✅         | 无图标注入，无文件复制                         |
| `generateRouter()`                   | ✅         | 读取 src/pages.json，输出 src/router.config.ts |
| `generateVersion()`                  | ✅         | 时间戳格式，输出 version.json                  |
| `htmlInject({ rules })`              | ⚠️         | 需提供注入规则                                 |
| `loadingManager()`                   | ✅         | 默认可见，DOMContentLoaded 自动隐藏            |
| `versionUpdateChecker()`             | ✅         | auto 模式，5 分钟检查，modal 提示              |

### 4.3 渐进式配置示例

```typescript
// 第一步：零配置启动
compressAssets()

// 第二步：覆盖关键参数
compressAssets({ algorithm: 'both' })

// 第三步：精细控制
compressAssets({
	algorithm: 'both',
	threshold: 2048,
	includeExtensions: ['.js', '.css', '.html'],
	excludePaths: ['assets/images'],
	compressionLevel: 9,
	brotliQuality: 11,
	reportOutput: 'reports/compress.json',
	parallelLimit: 5
})
```

### 6.2 BasePlugin — 生命周期与配置管理

v0.1.2 对 BasePlugin 进行了多项增强：

**三层配置合并**（v0.1.2 改用 `deepMerge`）：

```typescript
protected mergeOptions(options: T): Required<T> {
  const baseDefaults = { enabled: true, verbose: true, errorStrategy: 'throw' }
  const pluginDefaults = this.getDefaultOptions()
  return deepMerge(baseDefaults, pluginDefaults, options) as Required<T>
}
```

优先级：`baseDefaults` < `pluginDefaults` < `userOptions`。`deepMerge` 的关键语义：`undefined` 不覆盖（保护默认值），`null` 会覆盖（允许显式置空），嵌套对象递归合并，数组直接覆盖。

**v0.1.2 新增能力**：

| 能力                          | 说明                                                           |
| ----------------------------- | -------------------------------------------------------------- |
| `validator` 属性              | `Validator<T>` 实例，子类可在 `validateOptions` 中使用流式验证 |
| `getEnforce()` 方法           | 子类可覆盖以指定插件执行时机（如 `'post'`），默认 `undefined`  |
| `onConfigResolved(config)`    | Vite 配置解析完成后调用，存储 `viteConfig`                     |
| `handleError(error, context)` | 根据 `errorStrategy` 统一处理错误                              |
| `toPlugin()` 增强             | 自动注册 `configResolved` 和 `closeBundle` 钩子                |

**生命周期**：

```
constructor                    → 配置合并（deepMerge）· 日志初始化 · 配置验证（Validator）
    ↓
toPlugin()                     → 生成 Vite Plugin 对象
    ├── addPluginHooks()       → 子类注册业务钩子
    ├── configResolved         → 自动组合：基类存储 viteConfig → 子类钩子
    └── closeBundle            → 自动组合：子类清理资源 → 基类注销日志

```

### 6.3 Validator — 流畅 API 验证器

v0.1.2 的 Validator 支持完整的链式验证：

```typescript
this.validator
	.field('algorithm')
	.enum(['gzip', 'brotli', 'both'])
	.field('threshold')
	.number()
	.minValue(0)
	.field('includeExtensions')
	.array()
	.field('excludeExtensions')
	.array()
	.field('excludePaths')
	.array()
	.field('compressionLevel')
	.number()
	.minValue(1)
	.maxValue(9)
	.field('brotliQuality')
	.number()
	.minValue(1)
	.maxValue(11)
	.field('reportOutput')
	.custom(v => v === false || typeof v === 'string', '必须为 false 或字符串路径')
	.field('parallelLimit')
	.number()
	.minValue(1)
	.maxValue(50)
	.validate()
```

**核心特征**：延迟抛出 — 验证错误不立即抛出，收集到 `errors` 数组中，`validate()` 时一次性抛出所有错误。

**支持的方法**：

| 方法                                                           | 说明         |
| -------------------------------------------------------------- | ------------ |
| `field(name)`                                                  | 指定验证字段 |
| `required()`                                                   | 标记为必填   |
| `string()` / `boolean()` / `number()` / `array()` / `object()` | 类型验证     |
| `enum(values)`                                                 | 枚举验证     |
| `minValue(n)` / `maxValue(n)`                                  | 数值范围验证 |
| `default(value)`                                               | 设置默认值   |
| `custom(fn, msg)`                                              | 自定义验证   |
| `validate()`                                                   | 执行验证     |

### 9.4 自定义插件开发

```typescript
import { BasePlugin, createPluginFactory } from '@meng-xi/vite-plugin/factory'
import { injectBeforeTag, makeCallback } from '@meng-xi/vite-plugin/common'
import type { Plugin } from 'vite'

interface InjectSnippetOptions {
	targetTag?: string
	snippet: string
	onInjected?: string
}

class InjectSnippetPlugin extends BasePlugin<InjectSnippetOptions> {
	protected getPluginName(): string {
		return 'inject-snippet'
	}

	protected getDefaultOptions() {
		return { targetTag: '</body>' }
	}

	protected validateOptions() {
		this.validator.field('targetTag').string().field('snippet').required().string().validate()
	}

	protected addPluginHooks(plugin: Plugin): void {
		plugin.transformIndexHtml = {
			order: 'post',
			handler: (html: string) => {
				const result = injectBeforeTag(html, this.options.targetTag, this.options.snippet)
				if (result.injected) {
					this.logger.success('代码注入成功')
					return result.html
				}
				this.logger.warn('未找到目标标签，注入失败')
				return html
			}
		}
	}
}

export const injectSnippet = createPluginFactory(InjectSnippetPlugin)
```

## 十、compressAssets 深度解读

### 10.1 压缩流程

```
writeBundle 钩子触发
    ↓
scanDirectory(outDir, options)
    ├── 递归遍历 dist/ 目录
    ├── 对每个文件调用 shouldCompressFile()
    │   ├── threshold 过滤
    │   ├── excludeExtensions 过滤
    │   ├── includeExtensions 过滤
    │   ├── excludePaths 过滤
    │   └── 已压缩格式过滤（.gz / .br）
    └── 返回 FileCandidate[]
    ↓
按算法分组并发压缩
    ├── algorithm === 'gzip'  → compressFileGzip()
    ├── algorithm === 'brotli' → compressFileBrotli()
    └── algorithm === 'both'  → 先 gzip 后 brotli
    ↓
buildSummary() → 汇总统计
    ↓
可选：deleteOriginalFiles()
可选：writeReport()
logSummary() → Top 5 日志
```

### 10.2 跨平台路径兼容

路径比较时统一将反斜杠转换为正斜杠：

```typescript
const normalizedPath = relativePath.replace(/\\/g, '/')
const normalizedExclude = excludePath.replace(/\\/g, '/')
```

确保 Windows 和 Unix 系统下路径匹配行为一致。

### 10.3 并发压缩模型

`compressAssets` 使用 `runWithConcurrency` 实现 Worker Pool 并发控制：

```
任务队列: [file1, file2, file3, ..., fileN]
Worker Pool: min(parallelLimit, N) 个 Worker

Worker 1: file1 → file4 → file7 → ...
Worker 2: file2 → file5 → file8 → ...
Worker 3: file3 → file6 → file9 → ...

结果按原始索引存储，保证顺序一致性
```

### 10.4 与 Nginx 配合

```nginx
# nginx.conf
gzip_static on;       # 自动发送 .gz 预压缩文件
brotli_static on;     # 自动发送 .br 预压缩文件（需 brotli 模块）
```

当 Nginx 检测到请求的文件存在同名的 `.gz` 或 `.br` 文件时，会直接发送预压缩文件，跳过动态压缩，节省服务器 CPU。
