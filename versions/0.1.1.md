v0.1.0 建立了七插件体系与完整的开发框架，但有一个场景始终空缺——**构建时 HTML 内容注入**。你可能需要在 `<head>` 中注入 SEO meta 标签、第三方分析脚本、PWA 配置，或者根据环境变量动态插入不同内容。v0.1.1 的 `htmlInject`

## 二、htmlInject：构建时 HTML 内容注入

### 2.1 解决的问题

前端项目经常需要在 HTML 中注入各种内容：SEO meta 标签、第三方脚本、PWA manifest、环境变量驱动的动态内容。传统做法是手动编辑 `index.html`，但这种方式有三个问题：

1. **无法条件注入** — 开发环境和生产环境需要不同的内容，手动切换容易遗漏
2. **无法模板化** — 版本号、应用名称等动态值无法自动填充
3. **安全风险** — 直接拼接 HTML 字符串，容易引入 XSS 漏洞

### 2.2 核心设计

`htmlInject` 通过 `transformIndexHtml` 钩子在构建时注入内容，提供三层能力：

```
注入规则（InjectRule）
    │
    ├── 位置控制：head-start / head-end / body-start / body-end / selector
    ├── 条件注入：env / file-contains / custom
    └── 安全过滤：危险标签拦截 + 危险属性拦截 + 白名单机制
```

### 2.3 最小配置

```typescript
import { htmlInject } from '@meng-xi/vite-plugin'

export default defineConfig({
	plugins: [
		htmlInject({
			rules: [
				{
					id: 'meta-description',
					content: '<meta name="description" content="My Application">',
					position: 'head-end'
				}
			]
		})
	]
})
```

### 2.4 完整配置

```typescript
htmlInject({
	// 目标 HTML 文件
	targetFile: 'index.html', // 默认匹配所有 index.html

	// 注入规则数组
	rules: [
		{
			id: 'meta-description', // 规则标识符（日志用）
			content: '<meta name="description" content="{{appName}}">', // 支持模板变量
			position: 'head-end', // 注入位置
			priority: 10, // 优先级，数值越小越先执行
			templateVars: { appName: 'My App' } // 规则级模板变量
		},
		{
			id: 'analytics',
			content: '<script src="/analytics.js"></script>',
			position: 'body-end',
			condition: {
				type: 'env', // 条件类型：env / file-contains / custom
				value: 'PRODUCTION' // 环境变量名
			},
			allowScriptInjection: true // 允许注入脚本（跳过安全检查）
		},
		{
			id: 'replace-banner',
			content: '<div class="new-banner">Updated</div>',
			position: 'replace-selector', // 替换选择器匹配的内容
			selector: '<div class="old-banner">', // 选择器
			selectorMatch: 'string' // 'string' | 'regex'
		}
	],

	// 安全配置
	security: {
		blockDangerousTags: true, // 阻止 script、iframe、object 等
		blockDangerousAttributes: true, // 阻止 onclick、onload 等
		allowedTags: [], // 标签白名单
		blockedTags: [], // 自定义阻止标签
		blockedAttributes: [] // 自定义阻止属性
	},

	// 全局模板变量（可被规则级 templateVars 覆盖）
	templateVars: { version: '1.0.0' },

	// 日志输出
	logInjection: true
})
```

### 2.5 注入位置详解

| 位置               | 说明                  | 典型用途                |
| ------------------ | --------------------- | ----------------------- |
| `head-start`       | `<head>` 标签开始之后 | 早期加载的 CSS、preload |
| `head-end`         | `</head>` 标签之前    | meta 标签、第三方 SDK   |
| `body-start`       | `<body>` 标签开始之后 | 顶部导航、全局 banner   |
| `body-end`         | `</body>` 标签之前    | 分析脚本、统计代码      |
| `before-selector`  | 选择器匹配内容之前    | 在特定元素前插入        |
| `after-selector`   | 选择器匹配内容之后    | 在特定元素后插入        |
| `replace-selector` | 替换选择器匹配的内容  | 替换占位符、旧版内容    |

### 2.6 条件注入

三种条件类型覆盖不同场景：

```typescript
// 环境变量条件：仅在指定环境变量存在时注入
condition: { type: 'env', value: 'PRODUCTION' }

// 文件内容条件：仅当 HTML 包含指定字符串时注入
condition: { type: 'file-contains', value: '<div id="app">' }

// 自定义条件：函数返回 true 时注入
condition: { type: 'custom', value: (html, ctx) => html.includes('data-analytics') }

// 取反：条件不满足时注入
condition: { type: 'env', value: 'PRODUCTION', negate: true }
```

### 2.7 安全防护

`htmlInject` 内置了 XSS 防护机制，默认阻止危险标签和属性：

```
用户输入的 HTML 内容
    │
    ▼
┌──────────────────────────────────────┐
│  危险标签检测                         │
│  <script>、<iframe>、<object> 等      │
│  → 默认阻止，allowedTags 可放行       │
├──────────────────────────────────────┤
│  危险属性检测                         │
│  onclick、onload、onerror 等          │
│  → 默认阻止，blockedAttributes 可扩展 │
├──────────────────────────────────────┤
│  规则级放行                           │
│  allowScriptInjection: true           │
│  → 跳过安全检查，需确保内容可信        │
└──────────────────────────────────────┘
```

**何时使用 `allowScriptInjection`**：当你需要注入第三方脚本（如 Google Analytics、百度统计）且确认来源可信时。此选项会跳过该规则的安全检查，但全局安全配置仍然对其他规则生效。

## 三、PluginWithInstance：类型安全的插件实例访问

### 3.1 解决的问题

v0.1.0 中 `createPluginFactory` 返回的 Vite Plugin 对象上附带了 `pluginInstance` 属性，允许外部访问插件内部状态。但 TypeScript 无法推断这个属性的类型，导致 `pluginInstance` 是 `any` 类型。

### 3.2 使用方式

```typescript
import { generateVersion, loadingManager } from '@meng-xi/vite-plugin'
import type { PluginWithInstance } from '@meng-xi/vite-plugin/factory'
import type { GenerateVersionOptions } from '@meng-xi/vite-plugin/plugins/generate-version'
import type { LoadingManagerOptions } from '@meng-xi/vite-plugin/plugins/loading-manager'

export default defineConfig({
	plugins: [
		generateVersion({
			format: 'datetime',
			outputType: 'both'
		}) as PluginWithInstance<GenerateVersionOptions>,

		loadingManager({
			defaultVisible: true,
			autoHideOn: 'DOMContentLoaded'
		}) as PluginWithInstance<LoadingManagerOptions>
	]
})
```

通过 `as PluginWithInstance<XxxOptions>` 类型标注，你可以获得：

- `plugin.pluginInstance.options` 的完整类型提示
- `plugin.pluginInstance.logger` 的类型安全访问
- IDE 自动补全和类型检查

### 4.3 关键适配要点

**条件编译与平台判断**

uni-app 的条件编译（`#ifdef H5`）和 Vite 插件的 `enabled` 选项需要配合使用：

| 插件                   | 平台限制       | 原因                                  |
| ---------------------- | -------------- | ------------------------------------- |
| `buildProgress`        | 无             | 构建时工具，与运行平台无关            |
| `generateRouter`       | 无             | 读取 `pages.json`，与运行平台无关     |
| `generateVersion`      | 仅 H5 生产环境 | 版本检测依赖 `version.json` HTTP 请求 |
| `htmlInject`           | 仅 H5          | `transformIndexHtml` 仅 H5 平台生效   |
| `faviconManager`       | 仅 H5 生产环境 | 非 H5 平台无 HTML favicon 概念        |
| `copyFile`             | 仅 H5 生产环境 | 非 H5 平台由原生构建处理资源          |
| `loadingManager`       | 仅 H5          | 依赖 DOM API，非 H5 平台无 `window`   |
| `versionUpdateChecker` | 仅 H5 生产环境 | 依赖 `version.json` HTTP 请求         |

**HBuilderX 项目结构**

HBuilderX 创建的 uni-app 项目没有 `src/` 目录，所有源文件位于项目根目录：

```
uni-app/
├── App.vue              # 应用入口
├── main.js              # 主入口脚本
├── manifest.json        # 应用配置
├── pages.json           # 页面路由配置
├── pages/               # 页面目录
│   └── index/
│       └── index.vue
├── static/              # 静态资源
├── uni_modules/         # uni-app 插件
│   └── vite-plugin/     # @meng-xi/vite-plugin
├── index.html           # HTML 模板
└── vite.config.ts       # Vite 配置
```

因此 `generateRouter` 的 `pagesJsonPath` 应配置为 `'pages.json'`（而非 `'src/pages.json'`），`outputPath` 为 `'router.config.ts'`。

**LoadingManager 与 App.vue 协作**

`loadingManager` 配置了 `autoHideOn: 'DOMContentLoaded'`，但 uni-app 的 Vue 应用初始化发生在 DOMContentLoaded 之后。为确保应用完全就绪后再隐藏 Loading，需在 `App.vue` 的 `onLaunch` 中手动控制：

```vue
<script>
export default {
	onLaunch: function () {
		// #ifdef H5
		this.hideInitialLoading()
		// #endif
	},
	methods: {
		hideInitialLoading() {
			const manager = window.__LOADING_MANAGER__
			if (manager && manager.isVisible()) {
				manager.hide()
			}
		}
	}
}
</script>
```

### 4.4 全局变量类型声明

在 uni-app 项目中，`generateVersion` 注入的 `__APP_VERSION__`、`loadingManager` 注入的 `window.__LOADING_MANAGER__`、`versionUpdateChecker` 注入的 `window.__VUC_REFRESH__` 等全局变量需要类型声明。创建 `vite-env.d.ts`：

```typescript
/// <reference types="vite/client" />

interface VersionInfo {
	version: string
	buildTime: string
	timestamp: number
	format: string
	[key: string]: unknown
}

declare const __APP_VERSION__: string
declare const __APP_VERSION___INFO: VersionInfo

interface Window {
	__LOADING_MANAGER__?: import('./uni_modules/vite-plugin/js_sdk/plugins/loadingManager/index.js').LoadingManager
	__VUC_REFRESH__?: () => void
	__VUC_DISMISS__?: () => void
}
```

## 八、升级指南

### 从 v0.1.0 升级

```bash
pnpm add @meng-xi/vite-plugin@0.1.1 -D
```

无需修改任何现有配置。如需使用新功能：

1. **htmlInject** — 在 `plugins` 数组中添加 `htmlInject()` 配置
2. **PluginWithInstance** — 为需要的插件添加 `as PluginWithInstance<XxxOptions>` 类型标注
3. **uni-app 集成** — 参考[第四节](#四uni-app-全链路集成)的完整配置

### 从 v0.0.9 及更早版本升级
