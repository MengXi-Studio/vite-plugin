# 全局 Loading 状态管理：从白屏到请求拦截的完整方案 —— @meng-xi/vite-plugin v0.0.8

> 页面白屏时用户看到什么？请求并发时 Loading 闪烁怎么办？如何在不修改业务代码的前提下实现全局请求拦截？`@meng-xi/vite-plugin@0.0.8` 新增的 `injectLoading` 插件，为你提供从白屏 Loading 到请求自动拦截的一站式解决方案。

---

## 一、injectLoading 是什么？

`injectLoading` 是 `@meng-xi/vite-plugin` 的第六个内置插件，在构建阶段将全局 Loading 状态管理代码注入到应用中。它解决的核心问题是：

| 痛点                      | injectLoading 的解法                               |
| ------------------------- | -------------------------------------------------- |
| 白屏阶段无反馈            | CSS + HTML 静态注入到 `<head>`，HTML 解析即显示    |
| 请求并发时 Loading 闪烁   | 内置延迟显示、最小显示时间、防抖隐藏三重防闪烁机制 |
| 手动管理 Loading 状态繁琐 | 自动拦截 fetch/XHR 请求，按需显示/隐藏             |
| 不同项目样式需求各异      | 四种内置图标 + 完全自定义样式/模板                 |
| 插件回调无法传递函数      | 以函数体字符串形式提供，构建时注入浏览器端         |

### 与其他方案的对比

| 特性           | injectLoading | NProgress      | 自定义组件 |
| -------------- | ------------- | -------------- | ---------- |
| 白屏 Loading   | 原生支持      | 不支持         | 需额外实现 |
| 请求自动拦截   | fetch + XHR   | 仅路由级别     | 需手动封装 |
| 防闪烁机制     | 三重防护      | 仅最低显示时间 | 需自行实现 |
| 零业务代码侵入 | 是            | 否             | 否         |
| 自定义样式     | 完全可定制    | 有限           | 完全可定制 |
| 生命周期回调   | 5 个钩子      | 无             | 自定义     |

```typescript
import { defineConfig } from 'vite'
import { injectLoading } from '@meng-xi/vite-plugin'
```

## 三、核心场景详解

### 3.1 白屏 Loading — 首屏体验的关键

**问题**：SPA 应用在 JS 加载完成前，用户面对的是一片空白。即使有骨架屏，也需要等 JS 执行后才能渲染。

**injectLoading 的方案**：当 `defaultVisible` 为 `true` 时，CSS 样式和 HTML 结构以静态标签直接注入到 `<head>` 中，浏览器解析到 `<head>` 时 Loading 即可见，**无需等待任何 JS 执行**。

```typescript
injectLoading({
	defaultVisible: true, // DOM 初始可见
	autoHideOn: 'DOMContentLoaded' // DOM 解析完成后自动隐藏
})
```

**注入原理**：

```
<html>
  <head>
    <!-- inject-loading: head start -->
    <style data-loading-style>
      .__loading-overlay__ { position: fixed; ... }
      .__loading-visible__ { opacity: 1; visibility: visible; }
      .__loading-hidden__ { opacity: 0; visibility: hidden; }
    </style>
    <div class="__loading-overlay__ __loading-center__ __loading-visible__" id="__loading-root__">
      <div class="__loading-spinner__"></div>
      <div class="__loading-text__" data-loading-text="">加载中...</div>
    </div>
    <!-- inject-loading: head end -->
  </head>
  <body>
    <!-- 应用代码 -->
    <!-- inject-loading: body end -->
    <script>
      // Loading 管理器 JS 代码（IIFE）
      // 包含 show/hide/destroy 等运行时 API
      // 包含 autoHideOn 事件监听
    </script>
  </body>
</html>
```

**三种自动隐藏时机**：

| autoHideOn 值        | 触发条件          | 适用场景                          |
| -------------------- | ----------------- | --------------------------------- |
| `'DOMContentLoaded'` | DOM 解析完成      | SSR/MPA，页面结构简单             |
| `'load'`             | 所有资源加载完成  | 资源较重的页面（图片、字体等）    |
| `'manual'`           | 手动调用 `hide()` | Vue/React SPA，框架渲染完成后隐藏 |

```typescript
// Vue SPA 示例：在 app.mount() 后手动隐藏
injectLoading({
	defaultVisible: true,
	autoHideOn: 'manual'
})

// main.ts
const app = createApp(App)
app.mount('#app')
window.__LOADING_MANAGER__.hide()
```

### 3.2 请求自动拦截 — 零侵入的 Loading 管理

**问题**：每个异步请求都需要手动管理 Loading 状态，代码重复且容易遗漏。

**injectLoading 的方案**：通过 `autoBind` 配置自动拦截 fetch/XHR 请求，请求开始时自动显示，请求结束时自动隐藏。

```typescript
injectLoading({
	autoBind: 'all', // 拦截所有 fetch 和 XHR 请求
	requestFilter: {
		excludeUrlPrefixes: ['/static/', '/api/health', 'data:'],
		excludeMethods: ['OPTIONS', 'HEAD']
	}
})
```

**四种拦截模式**：

| autoBind 值      | 行为                  |
| ---------------- | --------------------- |
| `'none'`（默认） | 不拦截任何请求        |
| `'fetch'`        | 仅拦截 fetch 请求     |
| `'xhr'`          | 仅拦截 XMLHttpRequest |
| `'all'`          | 同时拦截 fetch 和 XHR |

**请求过滤机制**：

```
请求进入
  │
  ├─ excludeMethods 匹配？ ─── 是 ──→ 不触发 Loading
  │
  ├─ excludeUrlPrefixes 匹配？ ── 是 ──→ 不触发 Loading
  │
  ├─ includeUrls 非空？
  │     ├─ 是 ── URL 匹配 includeUrls？ ── 否 ──→ 不触发 Loading
  │     └─ 否 ── 继续
  │
  └─ excludeUrls 匹配？ ── 是 ──→ 不触发 Loading
        │
        └─ 否 ──→ 触发 Loading
```

> `includeUrls` 优先级高于 `excludeUrls`：当 `includeUrls` 非空时，仅匹配的 URL 才会触发 Loading。

**并发请求管理**：内部维护 `_pendingCount` 计数器，第一个请求开始时显示 Loading，最后一个请求结束时隐藏，中间的请求不会触发闪烁。

### 3.3 防闪烁三重机制

频繁的 Loading 显示/隐藏会导致视觉闪烁，injectLoading 提供了三层防护：

```
请求开始                              请求结束
  │                                     │
  │  ┌─────────────────┐               │  ┌─────────────────┐
  │  │  delayShow       │               │  │  minDisplayTime  │
  │  │  延迟 200ms 显示  │               │  │  至少显示 300ms   │
  │  │  请求在 200ms 内  │               │  │  确保用户能看到   │
  │  │  完成→不显示      │               │  │  Loading 动画     │
  │  └─────────────────┘               │  └─────────────────┘
  │                                     │
  │                                     │  ┌─────────────────┐
  │                                     │  │  debounceHide    │
  │                                     │  │  防抖 100ms 隐藏  │
  │                                     │  │  避免快速连续     │
  │                                     │  │  显示/隐藏        │
  │                                     │  └─────────────────┘
  │                                     │
  ▼                                     ▼
 show()                               hide()
```

```typescript
injectLoading({
	delayShow: { enabled: true, duration: 200 }, // 延迟显示
	minDisplayTime: { enabled: true, duration: 300 }, // 最小显示时间
	debounceHide: { enabled: true, duration: 100 } // 防抖隐藏
})
```

**典型场景**：

| 场景                                | 触发的机制     | 效果             |
| ----------------------------------- | -------------- | ---------------- |
| 请求 50ms 内完成                    | delayShow      | 不显示 Loading   |
| 请求 100ms 完成，Loading 显示 100ms | minDisplayTime | 继续显示至 300ms |
| 连续 5 个请求快速完成               | debounceHide   | 只触发一次隐藏   |

### 3.4 自定义样式 — 满足任何设计需求

**内置四种图标类型**：

| spinnerType         | 效果       | 适用场景       |
| ------------------- | ---------- | -------------- |
| `'spinner'`（默认） | 旋转圆环   | 通用场景       |
| `'dots'`            | 三点跳动   | 轻量级、现代感 |
| `'pulse'`           | 脉冲圆点   | 简约风格       |
| `'bar'`             | 滑动进度条 | 数据加载       |

**样式配置**：

```typescript
injectLoading({
	spinnerType: 'dots',
	style: {
		overlayColor: 'rgba(0, 0, 0, 0.6)', // 深色遮罩
		spinnerColor: '#ffffff', // 白色图标
		spinnerSize: '48px', // 更大的图标
		textColor: '#ffffff', // 白色文本
		textSize: '16px',
		zIndex: 10000, // 更高的层级
		pointerEvents: true, // 允许点击穿透
		backdropBlur: true, // 启用背景模糊
		backdropBlurAmount: 8 // 模糊程度
	},
	transition: {
		enabled: true,
		duration: 300,
		easing: 'ease-in-out'
	}
})
```

**完全自定义模板**：

当内置样式无法满足需求时，可以通过 `customTemplate` 提供完全自定义的 HTML：

```typescript
injectLoading({
	customTemplate: `
    <div class="my-loader">
      <svg class="my-spinner" viewBox="0 0 50 50">
        <circle cx="25" cy="25" r="20" fill="none" stroke="#4361ee" stroke-width="4"></circle>
      </svg>
      <span data-loading-text>加载中...</span>
    </div>
  `,
	style: {
		customClass: 'my-loading-theme',
		customStyle: 'backdrop-filter: blur(10px) saturate(180%)'
	}
})
```

> `customTemplate` 中必须包含一个具有 `data-loading-text` 属性的元素，否则 `updateText()` 方法将无法工作。

## 四、运行时 API 详解

`injectLoading` 在浏览器端注入 `window.__LOADING_MANAGER__`（可通过 `globalName` 自定义），提供以下 API：

```typescript
interface LoadingManager {
	show(text?: string): void // 显示 Loading，可选更新文本
	hide(): void // 隐藏 Loading（受 minDisplayTime 和 debounceHide 约束）
	forceHide(): void // 强制隐藏（忽略所有约束）
	updateText(text: string): void // 更新文本内容
	isVisible(): boolean // 查询当前是否显示
	getPendingCount(): number // 获取当前挂起的请求数
	destroy(): void // 销毁实例（清理 DOM、style、定时器、恢复拦截器）
}
```

### 使用示例

```typescript
// 自定义全局变量名
injectLoading({
	globalName: '__MY_LOADING__'
})

// 业务代码中使用
window.__MY_LOADING__.show('正在加载数据...')

try {
	await fetchData()
	window.__MY_LOADING__.updateText('正在处理...')
	await processData()
} finally {
	window.__MY_LOADING__.hide()
}

// 紧急场景：立即隐藏
window.__MY_LOADING__.forceHide()

// 查询状态
if (window.__MY_LOADING__.isVisible()) {
	console.log('Loading 正在显示')
}

// 清理（通常不需要手动调用，closeBundle 时自动执行）
window.__MY_LOADING__.destroy()
```

### 生命周期回调

回调以**函数体字符串**形式提供（构建时注入到浏览器端，无法传递函数引用）：

```typescript
injectLoading({
	callbacks: {
		onBeforeShow: 'console.log("即将显示")',
		onShow: 'console.log("已显示")',
		onBeforeHide: 'if (window.shouldKeepLoading) return false;',
		onHide: 'console.log("已隐藏")',
		onDestroy: 'console.log("已销毁")'
	}
})
```

- `onBeforeShow` / `onBeforeHide`：返回 `false` 可阻止显示/隐藏
- 所有回调自动包裹 try-catch，不会因回调错误影响主逻辑

## 五、技术实现深度解析

### 5.1 双阶段注入策略

injectLoading 采用 `<head>` + `<body>` 双阶段注入，这是实现白屏 Loading 的关键：

```
┌─────────────────────────────────────────────────────┐
│  <head> 阶段（defaultVisible: true 时注入）           │
│  ├── <style> CSS 样式                                │
│  └── <div> HTML 结构（初始可见）                      │
│  → 浏览器解析 <head> 时即渲染，零 JS 依赖             │
├─────────────────────────────────────────────────────┤
│  <body> 阶段（始终注入）                              │
│  └── <script> Loading 管理器 IIFE                    │
│  → 包含运行时 API、请求拦截、事件监听等               │
└─────────────────────────────────────────────────────┘
```

当 `defaultVisible` 为 `false` 时，CSS 和 HTML 由 JS 动态创建并插入 DOM，避免页面加载时出现不必要的 Loading。

### 5.2 请求拦截实现

**Fetch 拦截**：

```javascript
_originalFetch = window.fetch
window.fetch = function (input, init) {
	var url = typeof input === 'string' ? input : input instanceof URL ? input.href : input.url
	var method = (init && init.method) || (input && input.method) || 'GET'
	manager._requestStart(url, method)
	return _originalFetch.apply(this, arguments).then(
		function (response) {
			manager._requestEnd(url, method)
			return response
		},
		function (error) {
			manager._requestEnd(url, method)
			throw error
		}
	)
}
```

**XHR 拦截**：

```javascript
_originalXHROpen = XMLHttpRequest.prototype.open
_originalXHRSend = XMLHttpRequest.prototype.send

XMLHttpRequest.prototype.open = function (method, url) {
	this.__loadingUrl = url
	this.__loadingMethod = method
	return _originalXHROpen.apply(this, arguments)
}

XMLHttpRequest.prototype.send = function () {
	var self = this
	manager._requestStart(self.__loadingUrl, self.__loadingMethod)
	var _origLoadEnd = self.onloadend
	self.onloadend = function () {
		manager._requestEnd(self.__loadingUrl, self.__loadingMethod)
		if (_origLoadEnd) _origLoadEnd.apply(this, arguments)
	}
	return _originalXHRSend.apply(this, arguments)
}
```

> **关键设计**：XHR 拦截仅使用 `onloadend` 事件，而非同时监听 `onload`/`onerror`/`onabort`/`ontimeout`。`onloadend` 在所有终止场景后仅触发一次，避免了 `_requestEnd` 被多次调用导致计数器错误。

### 5.3 RegExp 序列化

`requestFilter` 中的 `excludeUrls` 和 `includeUrls` 是 `RegExp[]` 类型，但注入到浏览器端的代码是字符串，需要序列化处理：

```javascript
// 序列化：保存 source 和 flags
var _excludeUrls = JSON.stringify(excludeUrls.map(r => ({ source: r.source, flags: r.flags }))).map(function (s) {
	return new RegExp(s.source, s.flags)
})
```

这确保了正则表达式的 flags（如 `i`、`g`）在序列化/反序列化过程中不会丢失。

### 5.4 CSS 过渡与 JS 协同

CSS 中的 `transition` 属性根据 `transition` 配置项动态生成，与 JS 的 `_applyTransition` 方法协同工作：

```css
/* CSS 中根据配置生成 */
.__loading-hidden__ {
	opacity: 0;
	visibility: hidden;
	transition:
		opacity 200ms ease-out,
		visibility 200ms ease-out;
}
.__loading-visible__ {
	opacity: 1;
	visibility: visible;
	transition:
		opacity 200ms ease-out,
		visibility 200ms ease-out;
}
```

```javascript
// JS 中动态设置（与 CSS 一致）
function _applyTransition(show) {
	if (!_transition.enabled || !_loadingEl) return
	_loadingEl.style.transition = 'opacity ' + d + 'ms ' + e + ', visibility ' + d + 'ms ' + e
}
```

两者使用相同的 `duration` 和 `easing` 值，确保过渡效果一致且不会互相覆盖。

### 5.5 销毁机制

`destroy()` 方法确保所有资源被正确清理：

```
destroy()
  ├── _destroyed = true              // 标记已销毁，所有方法调用将被忽略
  ├── _clearTimers()                 // 清理所有定时器
  ├── 移除 loading DOM 元素           // _loadingEl.parentNode.removeChild()
  ├── 移除 style 元素                // _styleEl.parentNode.removeChild()
  ├── 重置所有状态变量                // _visible, _showTime, _pendingCount
  ├── _restoreInterceptors()         // 恢复原始 fetch/XHR
  │   ├── window.fetch = _originalFetch
  │   ├── XMLHttpRequest.prototype.open = _originalXHROpen
  │   └── XMLHttpRequest.prototype.send = _originalXHRSend
  └── _onDestroy()                   // 执行销毁回调
```

### 8.2 Bug 修复清单

| 问题                                   | 影响                                       | 修复方案                              |
| -------------------------------------- | ------------------------------------------ | ------------------------------------- |
| XHR 拦截多次调用 `_requestEnd`         | `_pendingCount` 计数错误，Loading 无法隐藏 | 仅使用 `onloadend` 事件               |
| `autoHideOn` 在事件已触发后不执行      | 白屏 Loading 永远不会自动隐藏              | 检测 `readyState`，已触发则立即执行   |
| RegExp 序列化丢失 flags                | `excludeUrls` 中的正则 flags 丢失          | 序列化时保存 `{ source, flags }`      |
| CSS 过渡与 JS 动态 transition 冲突     | 过渡动画异常                               | CSS transition 根据配置动态生成       |
| `show()` 重新显示时 `_showTime` 未重置 | `minDisplayTime` 计算错误                  | `show()` 时重置 `_showTime` 为 0      |

| `destroy()` 未恢复 XHR 拦截            | 请求拦截器无法恢复                         | 保存并恢复 `open`/`send` 原始引用     |
| 双重 try-catch 冗余                    | 代码冗余                                   | 移除外层冗余错误捕获                  |

## 十、注意事项与最佳实践

1. **白屏 Loading 与请求拦截不要同时使用 `defaultVisible: true` + `autoBind: 'all'`**：白屏阶段没有请求，两者独立工作。建议白屏 Loading 用 `autoHideOn` 控制，请求拦截用 `autoBind` 控制。

2. **`globalName` 需要合法的 JS 标识符**：因为它被用作 `window[globalName]` 的属性名，避免使用特殊字符。

3. **`callbacks` 是函数体字符串，不是函数引用**：构建时注入到浏览器端，无法传递闭包变量。如需访问运行时变量，请使用 `window` 上的全局变量。

4. **`destroy()` 通常不需要手动调用**：插件在 Vite `closeBundle` 钩子中自动调用 `destroy()`，仅在特殊场景（如热更新后重新初始化）需要手动调用。

5. **请求过滤优先使用 `excludeUrlPrefixes`**：前缀匹配比正则匹配更高效，对于固定前缀的 URL 应优先使用。

6. **`debounceHide` 在高并发场景下建议开启**：当多个请求几乎同时完成时，防抖可以避免 Loading 快速闪烁。

7. **SSR 环境自动安全**：注入的 JS 代码包含 `typeof window === 'undefined'` 检测，SSR 环境自动跳过，无需额外处理。
